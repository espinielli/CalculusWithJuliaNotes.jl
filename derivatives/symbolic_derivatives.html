24  Symbolic derivatives

We use the SymPy package in this section.

using SymPy

The ability to breakdown an expression into operations and their arguments is necessary when trying to apply the differentiation rules. Such rules are applied from the outside in. Identifying the proper “outside” function is usually most of the battle when finding derivatives. This example shows that computers arrange expressions in a manner that easily allows the outer function to be identified.

In the following example, we provide a sketch of a framework to differentiate expressions by a chosen symbol to illustrate how the outer function drives the task of differentiation.

The Symbolics package provides native symbolic manipulation abilities for Julia, similar to SymPy, though without the dependence on Python. The SymbolicUtils package, used by Symbolics, provides a generic interface for expression manipulation. Three main function verbs are istree (does an expression have sub-parts), operation (the “outer” function), and arguments (to describe the values passed to the outer function).

istree(x::Sym) = !(x.is_Atom)

function operation(x::Sym)
    nm = Symbol(SymPy.Introspection.funcname(x))
    ops = (Add=+, Mul=*, Pow=^, exp=exp, log=log, sin=sin, cos=cos)
    ops[nm]
end

arguments(x::Sym) =  collect(SymPy.Introspection.args(x))
arguments (generic function with 1 method)
Note

These are defined for SymPy objects when SymbolicUtils is loaded, but for illustration purposes we avoid that and define them directly above.

To see how the outer function is identified, we define a symbolic variable, a complicated expression, and call operation:

@syms x
ex = sin(x + cos(x^2))
outer = operation(ex)
sin (generic function with 20 methods)

The sin function is returned. (Following SymbolicUtils operation returns a function object, not the name of the function.)

The arguments are the terms that the outside function is called on. For our purposes there may be \(1\) (unary), \(2\) (binary), or more than \(2\) (nary) arguments. (We ignore zero-argument functions.) For example:

arguments(sin(x)), arguments(x^2), arguments(1 + x + x^2)
(Sym[x], Sym[x, 2], Sym[1, x, x^2])

(The last one may be surprising, but all three arguments are passed to the + function.)

Next, to define a function to decide the arity of an expression based on the number of arguments it is called with:

function arity(ex)
        n = length(arguments(ex))
        n == 1 ? Val(:unary) :
        n == 2 ? Val(:binary) : Val(:nary)
end
arity(1 + x + x^2)
Val{:nary}()

Differentiation must distinguish between expressions, variables, and numbers. Mathematically expressions have an “outer” function, whereas variables and numbers can be directly differentiated. The istree function in SymbolicUtils returns false when passed a symbol or numeric literal and true for more complicated expressions.

When this Dfunction encounters a complicated expression it dispatches to a specific method of D based on the outer operation and the arity of the expression. This may happen recursively. Otherwise, if the expression has no operation, D takes the derivative of the value – only the expression matching the variable will have non-zero derivative.

function D(ex, var=x)
    if istree(ex)
        outer, args = operation(ex), arguments(ex)
        return D(Val(outer), arity(ex), args, var)
    end

    return ex == x ? one(x) : zero(x)

end
D (generic function with 2 methods)

Now to develop methods for D for different “outside” functions and arities.

Addition can be binary, or nary. Here we implement the sum rule:

function D(::Val{+}, ::Val{:binary}, args, var)
    a, b = args
    a′, b′ = D(a,var), D(b,var)
    +(a′ + b′)
end

function D(::Val{+}, ::Val{:nary}, args, var)
    a′s = D.(args, var)
    +(a′s...)
end
D (generic function with 4 methods)

The binary case applies D to each argument (the “primes” are unicode notation achieved through \prime[tab] and not operations.) The nary case does something similar, only using broadcasting and splatting to produce the sum for arbitrary lengths.

Subtraction would also be implemented in a similar manner, though no nary case would be needed, a unary one might be:

function D(::Val{-}, ::Val{:unary}, args, var)
    a′ = D(first(args), var)
    -a′
end

function D(::Val{-}, ::Val{:binary}, args, var)
    a′, b′ = D.(args, var)
    a′ - b′
end
D (generic function with 6 methods)

(Technically, - is never an operation, as SymPy rewrites an expression like x - y as x + (-1 * y), so the above won’t be utilized.)

The product rule has the same cases as addition:

function D(::Val{*}, ::Val{:binary}, args, var)
    a, b = args
    a′, b′ = D(a, var), D(b, var)
    a′ * b + a * b′
end

function D(::Val{*}, ::Val{:nary}, args, var)
    a, bs... = args
    b = *(bs...)
    a′, b′ = D(a, var), D(b, var)
    a′ * b + a * b′
end
D (generic function with 8 methods)

The nary case above just peels off the first factor and then uses the binary product rule. A more efficient method could be developed.

Division is only a binary operation; a rule would use the quotient rule:

function D(::Val{/}, ::Val{:binary}, args, var)
    u, v = args
    u′, v′ = D(u, var), D(v, var)
    (u′ * v - u * v′) / v^2
end
D (generic function with 9 methods)

(As with -, division is not part of SymPy’s internal representation, so this definition won’t be encountered.)

Powers are handled a bit differently. The power rule would require checking if the exponent does not contain the variable of differentiation, exponential derivatives would require checking the base does not contain the variable of differentation. Trying to implement both would be tedious. The following uses the fact that \(x = \exp(\log(x))\) (for x in the domain of log, more care is necessary if x is negative) to differentiate:

function D(::Val{^}, ::Val{:binary}, args, var)
    a, b = args
    u = sympy.exp(b*log(a), evaluate=false)
    D(u, var)  # a > 0 assumed here
end
D (generic function with 10 methods)

The evaluate=false part is technical, suppressing any simplification that might rewrite the expression using ^.

That leaves the task of defining a rule to differentiate both exp and log. This is done, along with sin and cos with unary definitions.

function D(::Val{exp}, ::Val{:unary}, args, var)
    a = only(args)
    a′ = D(a, var)
    exp(a) * a′
end

function D(::Val{log}, ::Val{:unary}, args, var)
    a = only(args)
    a′ = D(a, var)
    1/a * a′
end

function D(::Val{sin}, ::Val{:unary}, args, var)
    a = only(args)
    a′ = D(a, var)
    cos(a) * a′
end

function D(::Val{cos}, ::Val{:unary}, args, var)
    a = only(args)
    a′ = D(a, var)
    -sin(a) * a′
end
D (generic function with 14 methods)

The pattern is similar for each. The a′ factor is needed due to the chain rule. The above illustrates the simple pattern necessary to add a derivative rule for a function.

Note

Several automatic differentiation packages use a set of rules defined following an interface spelled out in the package ChainRules.jl. Leveraging multi-dimensional derivatives, the chain rule is the only rule needed of the sum, product, quotient and chain rules.

More functions could be included, but for this example the above will suffice, as now the system is ready to be put to work.

ex = x + 2/x
D(ex, x)
\[ 1 - \frac{2}{x^{2}} \] 
ex = (x + sin(x)) / sin(x)
D(ex, x)
\[ - \frac{\left(x + \sin{\left(x \right)}\right) \cos{\left(x \right)}}{\sin^{2}{\left(x \right)}} + \frac{\cos{\left(x \right)} + 1}{\sin{\left(x \right)}} \] 
ex = exp(log(x) + sin(x) + x^2) / x
D(ex, x)
\[ \left(2 x + \cos{\left(x \right)}\right) e^{x^{2} + \sin{\left(x \right)}} \]

Finally, we have a second derivative taken below:

ex = sin(x) - x - x^3/6
D(D(ex, x), x)
\[ - x - \sin{\left(x \right)} \]
